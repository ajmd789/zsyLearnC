zsyLearnC 在 Debian Dev Container 中集成 gtest/gmock 与稳定性测试

这份文档不是泛泛讲测试框架，而是直接针对你当前这个仓库：CMake + C++17 + gRPC + Debian 13 Dev Container。重点回答 4 个问题：gtest 能不能 mock、为什么 doctest 不够、你这个项目先改哪里、程序跑起来以后怎么验证稳定性。

文档基线时间：2026-03-08。版本信息以 Debian 13 stable（trixie）官方包页面为准；如果你的容器已经做过镜像更新，最终以容器内 apt-cache policy 为准。

项目: zsyLearnC 容器: Debian 13 测试框架: GoogleTest / GoogleMock 目标: mock + 稳定性

1. 先给结论

1.1 gtest 能 mock 吗

严格说，GoogleTest 负责断言、测试运行和报告；GoogleMock 负责 mock。它们属于同一套 googletest 项目，工程里一般一起用。

1.2 doctest 能 mock 吗

doctest 本身不是 mock 框架。你可以把它和别的 mocking library 配合，但如果你已经明确需要 mock，并且要做线程/失败路径/稳定性验证，直接切到 gtest + gmock 更省事。

本项目的关键判断：
你当前仓库里最值得先 mock 的不是 gRPC 本身，而是 GrpcMessageClient、睡眠行为和时间来源。否则单元测试会慢、脆、难覆盖失败路径。
问题 建议 原因
只想写断言，不需要 mock doctest 也能用 轻量、上手快
需要 mock 依赖、验证调用顺序、造失败场景 用 gtest + gmock 官方组合成熟，和 CMake/CTest 集成顺手
要关注程序跑起来后的稳定性 单元测试 + 集成测试 + sanitizer + soak test 一起做 只做 mock 不能发现内存、线程、长时间运行问题

2. 当前仓库现状

从现有代码看，你的主干逻辑集中在这些文件：

  • src/app/pingpong_app.cpp：整体编排，固定端口、固定轮数、固定 5 秒间隔。
  • src/common/grpc_message_client.cpp：真实 gRPC client，直接发 RPC，失败时只记日志。
  • src/common/messenger_service.cpp：真实 gRPC server handler。
  • src/node1/node1_sender.cpp、src/node2/node2_sender.cpp：线程发送逻辑。
  • src/common/turn_coordinator.cpp：线程间轮次协调。
当前实现 对测试的影响 建议
Node1Sender/Node2Sender 直接依赖具体类 GrpcMessageClient 没法用 gmock 替换依赖，单元测试难写 抽 IMessageClient 接口
代码里直接 std::this_thread::sleep_for(interval_) 测试真的要等 5 秒，速度非常差 抽 ISleeper 或传入 sleep 回调
CurrentTimeString() 是自由函数 日志内容难稳定断言 抽 ITimeProvider 或别在单元测试里直接断完整日志
端口和轮数写死在 PingPongApp::Run() 集成测试容易撞端口，也不方便压测 引入 AppConfig
Send() 失败只返回 false，上层未处理 失败路径行为不明确 明确“重试/终止/告警”策略并补测试
一个很重要的现实：
gmock 并不能“直接” mock 任意 C++ 代码。它最擅长 mock 虚函数接口。像自由函数、构造函数、非虚成员函数、全局单例，通常都要先加一层 adapter 或接口。

4. Dev Container 里先做什么

5. 先抽哪些接口才能 mock

这里不要从 PingPongApp 整体开始 mock。先从发送线程逻辑入手，收益最大。

5.1 第一个接口：消息发送客户端

class IMessageClient {
 public:
  virtual ~IMessageClient() = default;
  virtual bool Send(const std::string& from,
                    const std::string& to,
                    int sequence) = 0;
};

让现有的 GrpcMessageClient 实现这个接口，然后把 Node1Sender 和 Node2Sender 的构造函数参数从具体类改成接口引用。

class GrpcMessageClient : public IMessageClient {
 public:
  GrpcMessageClient(const std::string& target, ThreadSafeLogger& logger);
  bool Send(const std::string& from,
            const std::string& to,
            int sequence) override;
};

5.2 第二个接口：睡眠行为

class ISleeper {
 public:
  virtual ~ISleeper() = default;
  virtual void SleepFor(std::chrono::milliseconds duration) = 0;
};

原因很直接：你现在的发送线程每轮都真睡 5 秒。如果不把 sleep 抽出来，单元测试会慢得不可接受。

5.3 第三个接口：时间来源

这一步不是第一优先级，但如果你后面要认真测日志、超时、重试，最好把 CurrentTimeString() 包成时间提供者。

5.4 第四个对象：App 配置

struct AppConfig {
  std::string node1_server_addr = "0.0.0.0:50051";
  std::string node2_server_addr = "0.0.0.0:50052";
  std::string node1_target = "127.0.0.1:50051";
  std::string node2_target = "127.0.0.1:50052";
  int rounds = 10;
  std::chrono::milliseconds interval{5000};
  std::chrono::milliseconds startup_wait{500};
};
为什么这一步重要：
单元测试要把 interval 设成 0ms；集成测试要把端口改成测试专用端口；稳定性测试要把 rounds 放大，甚至支持无限循环。这些都不应该硬编码在 PingPongApp::Run() 里。

5.5 mock 示例

#include <gmock/gmock.h>

class MockMessageClient : public IMessageClient {
 public:
  MOCK_METHOD(bool, Send,
              (const std::string&, const std::string&, int),
              (override));
};

class MockSleeper : public ISleeper {
 public:
  MOCK_METHOD(void, SleepFor, (std::chrono::milliseconds), (override));
};

6. CMake 接入步骤

建议不要把所有生产代码都直接扔给测试目标。先把“可单元测试的逻辑”和“真实 runtime/gRPC 组装”拆开。

6.1 建议的目标拆分

目标 内容 作用
pingpong_proto protobuf/gRPC 生成代码 现有保留
pingpong_logic TurnCoordinator、Node1Sender、Node2Sender 等纯逻辑 供单元测试链接
pingpong_runtime GrpcMessageClient、MessengerService、server_utils、PingPongApp 真实运行与集成测试

7. 第一批测试怎么写

先别碰最复杂的 PingPongApp。第一批测试的目标是验证最容易出错、又最容易 mock 的逻辑。

7.1 先写这 4 个

  1. TurnCoordinator：初始轮次是否正确，SetTurn() 后等待方是否被唤醒。
  2. Node1Sender：拿到轮次后是否调用 Send("node1", "node2", round)。
  3. Node2Sender：最后一轮是否把轮次切成 Done。
  4. 发送失败路径：如果 Send() 返回 false，程序下一步到底要继续、停止还是计数告警。

7.2 一个典型的 gmock 测试样子

using ::testing::InSequence;
using ::testing::Return;

TEST(Node1SenderTest, SendsCurrentRoundToNode2) {
  MockMessageClient client;
  MockSleeper sleeper;
  common::TurnCoordinator coordinator(common::Turn::Node1ToNode2);

  node1::Node1Sender sender(client, sleeper, coordinator, 1,
                            std::chrono::milliseconds(0));

  {
    InSequence seq;
    EXPECT_CALL(client, Send("node1", "node2", 1))
        .WillOnce(Return(true));
    EXPECT_CALL(sleeper, SleepFor(std::chrono::milliseconds(0)));
  }

  sender.Run();
  EXPECT_EQ(coordinator.WaitFor(common::Turn::Node2ToNode1),
            common::Turn::Node2ToNode1);
}
如果你发现这个测试写不出来，不是 gmock 有问题，而是你的生产代码还没被设计成“可测试”。先改设计，再写测试。

8. 集成测试怎么做

单元测试只解决“某个类逻辑对不对”。而你这个项目是 gRPC + 线程 + server/client 编排，至少还需要一层真实组件测试。

8.1 建议的集成测试目标

  • 真实启动 MessengerService 和 GrpcMessageClient，验证一轮 RPC 成功。
  • 验证 server 正常启动和正常 shutdown，不残留线程。
  • 验证错误端口、server 未启动时，client 是否能返回失败并记日志。

8.2 这里要改的两个点

  1. 不要把测试端口写死成 50051/50052。否则本地同时跑主程序和测试会撞端口。
  2. 最好支持动态端口，或者至少让测试显式传入专用端口。

8.3 推荐目录

tests/
  unit/
  integration/
  soak/
  mocks/
  support/

8.4 一个更实用的判断标准

对这个项目来说：

  • 单元测试不要真的开 gRPC server。
  • 集成测试要开真实 gRPC server，但轮数和间隔要压到最小。
  • 稳定性测试才允许长时间运行。

9. 稳定性测试怎么开展

“程序跑起来以后稳定”这件事，不能只靠 mock。你至少要把验证分成 5 层。

层次 目的 工具 / 方法 建议频率
单元测试 逻辑正确、失败路径明确 gtest + gmock 每次改代码都跑
集成测试 真实 gRPC 链路可用 CTest + 真实 server/client 每次提交前跑
Sanitizer 抓内存越界、未定义行为、线程竞争 ASan / UBSan / TSan 每日或每次大改后跑
Valgrind 补看泄漏和错误访问 Memcheck / Helgrind / DRD 里程碑前跑
Soak / Stress 长时间运行是否稳定 循环执行、监控 RSS/线程数/错误率 夜间任务

9.1 建议先加可压测参数

  • --rounds
  • --interval-ms
  • --startup-wait-ms
  • --node1-port、--node2-port
  • 可选：--forever 或 --duration-sec

9.2 ASan / UBSan 建议命令

cmake -S /workspace -B /workspace/build-asan -G Ninja \
  -DBUILD_TESTING=ON \
  -DCMAKE_BUILD_TYPE=Debug \
  -DCMAKE_C_FLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer" \
  -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer"

cmake --build /workspace/build-asan -j"$(nproc)"
ctest --test-dir /workspace/build-asan --output-on-failure

9.3 TSan 建议单独建目录

cmake -S /workspace -B /workspace/build-tsan -G Ninja \
  -DBUILD_TESTING=ON \
  -DCMAKE_BUILD_TYPE=Debug \
  -DCMAKE_C_FLAGS="-fsanitize=thread -fno-omit-frame-pointer" \
  -DCMAKE_CXX_FLAGS="-fsanitize=thread -fno-omit-frame-pointer"

cmake --build /workspace/build-tsan -j"$(nproc)"
ctest --test-dir /workspace/build-tsan --output-on-failure
注意： ASan/UBSan 和 TSan 不要混在一个构建目录里。线程 sanitizer 要单独跑，否则结果会很乱。

9.4 Valgrind 建议命令

cmake -S /workspace -B /workspace/build-valgrind -G Ninja \
  -DBUILD_TESTING=ON \
  -DCMAKE_BUILD_TYPE=RelWithDebInfo

cmake --build /workspace/build-valgrind -j"$(nproc)"

valgrind --leak-check=full --show-leak-kinds=all --track-origins=yes \
  /workspace/build-valgrind/pingpong_demo

9.5 Soak test 最少盯这几个指标

  • 进程是否异常退出。
  • 错误日志是否逐渐增多。
  • RSS 内存是否持续增长。
  • 线程数是否异常增加。
  • 端口是否释放干净，反复重启会不会失败。

9.6 建议的稳定性场景

场景 A: 1 轮、0ms 间隔，确认功能最小闭环 场景 B: 1000 轮、0ms 间隔，放大并发切换问题 场景 C: 反复启动/停止 100 次，找 shutdown/资源释放问题 场景 D: client 指向错误端口，验证失败路径 场景 E: 运行中途主动关闭一侧 server，观察恢复或退出行为 场景 F: 夜间长跑 30 分钟 / 2 小时，盯内存和线程数

10. 常见坑

10.1 “gmock 不能 mock 我的类”

通常不是 gmock 不行，而是你的类没有虚接口，或者依赖是自由函数/静态函数/构造行为。处理方式是先加一层接口或 adapter。

10.2 “测试非常慢”

根因大概率是你在测试里真的睡了 5 秒。把 sleep 抽出来，单元测试统一改成 0ms。

10.3 “测试不稳定，有时过有时不过”

多半是线程时序没控住。写 gmock 时，先设 EXPECT_CALL，再启动线程。不要倒过来。

10.4 “集成测试端口冲突”

不要复用 50051/50052。测试必须支持自定义端口，最好是动态端口。

10.5 “Send 失败后要不要继续”

这不是测试问题，是产品语义问题。你要先定规则，再写测试。最少要回答：

  • 失败后直接退出，还是继续下一轮？
  • 要不要重试？重试几次？间隔多久？
  • 是否需要把失败计入统计并输出最终摘要？

10.6 “只跑单元测试，一切都绿，但线上还是抖”

这是正常现象。mock 保证不了真正的线程竞争、gRPC 行为、内存泄漏和长时间运行问题，所以一定要补 sanitizer、Valgrind 和 soak test。

11. 推荐推进顺序

  1. 先补容器依赖，重建 Dev Container。
  2. 把发送端依赖抽成 IMessageClient。
  3. 把睡眠行为抽成 ISleeper。
  4. 接入 BUILD_TESTING、CTest、GTest::gmock。
  5. 先写 TurnCoordinator、Node1Sender、Node2Sender 的单元测试。
  6. 再给 PingPongApp 加配置对象，解除端口/轮数硬编码。
  7. 补一组真实 gRPC 集成测试。
  8. 最后再做 ASan/TSan/Valgrind/长跑脚本。
最务实的目标不是“今天把所有测试都补完”，而是先把仓库变成一个 能持续扩测试 的结构。第一步一定是把可 mock 的边界做出来。

13. 容器内 Soak Test 运行与资源观测

这一章专门回答两个现实问题：这里到底怎么跑，以及 ASan/TSan 要不要打进日常长跑。结论先说在前面：不要把 ASan/TSan 当成常规 soak 二进制。它们适合做“找 bug 的短跑或中跑”，不适合做“看真实 CPU/RSS 基线的长跑”。

13.1 先明确一件事：当前程序还不是真正的 soak 目标

你现在的 pingpong_demo 写死了 10 轮，并且 node1/node2 每轮都各自睡 5s。也就是说单次运行大约会持续 100s 左右，然后自然退出。所以：

  • 如果你现在不改代码，只能做“重复运行型稳定性测试”。
  • 如果你要做夜间长跑，最好先把程序改成支持 --rounds、--interval-ms、--forever 这类参数。

13.2 日常 soak 要不要把 ASan/TSan 打包进去

构建类型 是否建议用于长跑 建议用途
Release / RelWithDebInfo 建议 看真实 CPU、RSS、线程数、端口释放、长时间运行稳定性
ASan + UBSan 不建议做资源基线长跑 适合跑 5 分钟、10 分钟、几百轮这类“抓内存/未定义行为”验证
TSan 不建议做夜间资源长跑 适合专门抓数据竞争；开销大，结果会明显扭曲 CPU/内存表现
结论：
你应该至少保留 3 套构建目录：build-soak、build-asan、build-tsan。build-soak 看资源曲线；build-asan 和 build-tsan 负责找问题，不负责给出真实资源基线。

13.3 容器里先补观测工具

下面这些命令都针对 Debian 容器内执行：

apt-get update && apt-get install -y --no-install-recommends \
  procps \
  psmisc \
  iproute2 \
  lsof \
  sysstat \
  && rm -rf /var/lib/apt/lists/*
工具 用途
ps / top 看进程 CPU、内存、线程数
pidstat 持续记录某个 PID 的 CPU、RSS、页错误、I/O

pstree 看进程树

13.5 如果你现在先不改代码：用“重复运行型” soak

这条路径适合当前仓库原样验证。因为程序单次运行就会退出，所以你只能连续跑很多次，盯有没有崩、有没有端口没释放、有没有逐次变慢。

for i in $(seq 1 100); do
  ts=$(date '+%F %T')
  echo "=== run ${i} ${ts} ===" | tee -a /workspace/soak/logs/repeat.log

  /workspace/build-soak/pingpong_demo \
    >> /workspace/soak/logs/app-repeat.log 2>&1 &
  APP_PID=$!
  echo "${APP_PID}" >> /workspace/soak/logs/pids.log

  pidstat -r -u -d -h -p "${APP_PID}" 5 \
    >> /workspace/soak/metrics/pidstat-repeat.log 2>&1 &
  PIDSTAT_PID=$!

  while kill -0 "${APP_PID}" 2>/dev/null; do
    ts=$(date '+%F %T')
    echo "[${ts}]" >> /workspace/soak/metrics/ps-repeat.log
    ps -o pid,ppid,%cpu,%mem,rss,vsz,nlwp,etime,stat,cmd -p "${APP_PID}" \
      >> /workspace/soak/metrics/ps-repeat.log
    awk '/VmRSS|VmSize|Threads/ {print}' /proc/"${APP_PID}"/status \
      >> /workspace/soak/metrics/status-repeat.log
    ss -lntp | grep -E '50051|50052' \
      >> /workspace/soak/metrics/ports-repeat.log || true
    sleep 5
  done

  wait "${APP_PID}"
  RC=$?
  kill "${PIDSTAT_PID}" 2>/dev/null || true

  echo "run=${i} rc=${RC}" >> /workspace/soak/logs/repeat-exit.log
  ss -lntp | grep -E '50051|50052' \
    >> /workspace/soak/metrics/ports-after-exit.log || true

  if [ "${RC}" -ne 0 ]; then
    echo "repeat soak stopped at run ${i}" | tee -a /workspace/soak/logs/repeat.log
    break
  fi
done
这条命令链最适合对应 9.6 的场景 A、B、C、D。你可以把 seq 1 100 改成更大的次数，观察第几十次以后是否开始报错、端口冲突、退出码异常。

13.6 推荐的长跑方式：先把程序改成支持参数

真正像样的 soak，不应该靠 shell 循环不断重启，而应该让同一个进程长时间跑。推荐至少支持这些参数：

  • --rounds 1000000
  • --interval-ms 0
  • --startup-wait-ms 50
  • --node1-port 50051、--node2-port 50052
  • 可选：--forever 或 --duration-sec 7200

参数化以后，长跑命令可以像这样：

/workspace/build-soak/pingpong_demo \
  --rounds 1000000 \
  --interval-ms 0 \
  --startup-wait-ms 50 \
  --node1-port 50051 \
  --node2-port 50052 \
  > /workspace/soak/logs/app-soak.log 2>&1 &

APP_PID=$!
echo "${APP_PID}" > /workspace/soak/logs/soak.pid

13.7 同时采集 CPU、RSS、线程数、端口

pidstat -r -u -d -h -p "${APP_PID}" 5 \
  > /workspace/soak/metrics/pidstat-soak.log 2>&1 &
PIDSTAT_PID=$!

while kill -0 "${APP_PID}" 2>/dev/null; do
  ts=$(date '+%F %T')

  echo "[${ts}]" >> /workspace/soak/metrics/ps-soak.log
  ps -o pid,ppid,%cpu,%mem,rss,vsz,nlwp,etime,stat,cmd -p "${APP_PID}" \
    >> /workspace/soak/metrics/ps-soak.log

  echo "[${ts}]" >> /workspace/soak/metrics/status-soak.log
  awk '/VmRSS|VmSize|Threads/ {print}' /proc/"${APP_PID}"/status \
    >> /workspace/soak/metrics/status-soak.log

  echo "[${ts}]" >> /workspace/soak/metrics/ports-soak.log
  ss -lntp | grep -E '50051|50052' \
    >> /workspace/soak/metrics/ports-soak.log || true

  if [ -f /sys/fs/cgroup/memory.current ]; then
    echo "${ts} $(cat /sys/fs/cgroup/memory.current)" \
      >> /workspace/soak/metrics/cgroup-memory.log
  fi

  if [ -f /sys/fs/cgroup/cpu.stat ]; then
    echo "[${ts}]" >> /workspace/soak/metrics/cgroup-cpu.log
    cat /sys/fs/cgroup/cpu.stat \
      >> /workspace/soak/metrics/cgroup-cpu.log
  fi

  sleep 5
done

wait "${APP_PID}"
RC=$?
kill "${PIDSTAT_PID}" 2>/dev/null || true
echo "rc=${RC}" > /workspace/soak/logs/soak-exit.code

13.8 跑完以后看什么

文件 看点
app-soak.log / app-repeat.log 有没有 Segmentation fault、terminate called、gRPC failed、超时、异常退出
pidstat-*.log CPU 是否持续飙高、RSS 是否持续上升、是否出现异常页错误/I/O
status-*.log Threads 是否单调增长，VmRSS 是否只涨不回
ports-*.log 进程退出后端口是否仍被占用
*-exit.code 退出码是否始终为 0

13.9 我建议你给 soak 设一个最小通过标准

  • 退出码始终为 0。
  • 日志中不出现连续增长的 gRPC failed/异常栈。
  • VmRSS 和 RSS 不出现无上限单调增长。
  • Threads 长时间保持稳定，不出现一轮一轮上涨。
  • 进程退出后 5 秒内，50051/50052 不再被监听。

13.10 ASan/TSan 应该怎么配合这套流程

  1. 先用 build-asan 跑短场景：1 轮、100 轮、1000 轮，找越界、UAF、未定义行为。
  2. 再用 build-tsan 跑 0ms 间隔的高频场景，专门看 race。
  3. 最后用 build-soak 跑 30 分钟、2 小时的长跑，观察真实资源曲线。
正确顺序不是“把 sanitizer 打进同一套包里一直跑”，而是“先 sanitizer 找 bug，再 release/relwithdebinfo 做真实长跑”。这两类目标不要混。
